  A few things I found worth writing down, since none of them are guessable from
  a quick look at the tree:

  The x64-only constraint. The solution advertises Win32/x86 configurations, but
  those ItemDefinitionGroups set neither the include directories nor C++20.
  Since every include in the codebase is flat (#include "Mtx4.h" with ~20
  directories on the include path), an x86 build cannot resolve a single header.
  A future instance debugging a build failure could easily waste time switching
  platforms, so I called that out explicitly.

  The threading model is the architecture. A logic thread at fixed 60 Hz and an
  unthrottled render thread share exactly one object — an SPSC triple buffer of
  LogicState — and the renderer interpolates between two logic states using
  timestamps carried in the state itself. The practical consequence, which I
  highlighted: any new time-varying Scene field must be added to Scene::Slerp or
  it will visibly stutter.

  What's stubbed versus what's real. This mattered enough to get its own
  section. There is no depth buffer despite depthbufferHandle being created and
  resized; ResourceManager::meshRegister is a @todo returning {0,0}, making the
  mesh handles at the top of Renderer.h meaningless; and the entire
  Backend/RenderCommand/ tree compiles but is dead — its only call site is
  commented out. Someone reading those files cold would reasonably assume
  they're load-bearing.

  I also noted the Czech comments and commit convention, and that _old/ is a
  prior engine iteration to leave alone.

  Two things I documented as intentional but that you may want to reconsider —
  flagging them rather than acting:

  - renderFrame rebuilds several meshes from MeshFactory every frame (the
  animated grid, icosphere, and axis cubes). Given that mesh registration is
  stubbed, this looks like a placeholder rather than a choice.
  - Renderer::resize assigns targetResize's return value straight back onto the
  handle, and that function returns TARGET_INVALID on failure — so one failed
  resize would permanently destroy the framebuffer handle.

  Hlavní příčina: lastLogicTick je nástěnný čas, ne simulační

  Tohle je podle mě jádro záseků. V TimeManager::dispatchAction
  (TimeManager.h:123-129):

  while (m_accumulator >= m_fixedDelta) {
      stepCallback(m_fixedDelta, sinceStart());   // ← čerstvé QPC, ne ideální 
  čas ticku
      m_accumulator -= m_fixedDelta;
  }

  sinceStart() je okamžik, kdy se callback náhodou spustil. Takže razítka jsou
  t_k = k·Δ + jitter_k, ne k·Δ. Renderer pak počítá alfu (Renderer.h:206-220):

  double stateDelta = timeCurr - timePrev;          // Δ ± jitter
  t = (visualTime - timePrev) / stateDelta;         // jitter v čitateli i 
  jmenovateli

  Jitter scheduleru se tím propíše přímo do interpolačního koeficientu. Alfa pak
  neroste lineárně s reálným časem → objekty se pohybují nerovnoměrně i při 
  vysokém FPS. To je přesně ten typ mikro-záseku, který nejde vysvětlit nízkým
  FPS. Ty výpisy "Zaskub: t = ..." na Renderer.h:222 jsou symptom tohohle.

  Fix je malý: v App si držet monotónní simulační hodiny (simTime += fixedDelta
  každý krok) a publikovat je. stateDelta je pak přesně Δ a alfa jede lineárně.
  Zároveň si všimni, že razítko se bere před updateLogic(), ale publikuje se
  stav po něm — stav v čase T+Δ dostane razítko T.

  Dokud tohle platí, std::clamp(t, 0.0, 1.1) na Renderer.h:223 problém jen
  maskuje (a ta 1.1 navíc nesymetricky extrapoluje).

  Druhá příčina: renderer alokuje jako o život

  renderFrame staví meše od nuly každý snímek (Renderer.h:128-149):

  - CreateGridWave(60, ...) → 3 600 vertexů se sqrt+sin, 6 962 trojúhelníků, 20
  886 indexů — a indexBuffer nemá reserve() (MeshFactory.cpp:317), takže roste
  realokacemi.
  - CreateIcosphere(1.0f, 4) → 20·4⁴ = 5 120 trojúhelníků, a
  MeshModifier::Subdivide si pro každou ze 4 iterací alokuje nový unordered_map
  + vector (MeshModifier.h:14-41).
  - k tomu 5× CreateCube s applyTransformation.

  A pak Slerp — každý snímek. Scene sceneInterpolated = b; je hluboká kopie 5
  mešů, Slerp(a.car, b.car, t) kopíruje CarTransformation (dalších 5 mešů), a
  WheelTransformation::Lerp kopíruje mesh 4×. To je ~10 hlubokých kopií mešů na 
  snímek, na render vlákně. Při 1000 FPS to jsou desítky tisíc malloc/free za
  sekundu.

  Ta struktura se jmenuje ZeroAllocTripleBuffer, ale T = LogicState obsahuje
  Scene s Meshi, takže zero-alloc je jen ta indexová výměna — payload alokuje na
  obou vláknech (scenePublish na logice, advance_and_load_current na
  rendereru). Meše ve Scene jsou navíc neměnné — nemají tam co dělat, patří do
  registry vedle handle.

  Triple buffer: skutečný lost-update race

  update_reader() (ZeroAllocTripleBuffer.h:35-48) resetuje has_new_data až po
  výměně:

  P: exchange(dirty) ← A;  P: has_new_data = true
  C: load → true
  C: exchange(dirty) → dostane A
  P: exchange(dirty) ← B;  P: has_new_data = true
  C: has_new_data = false        ← přepíše producentovo true; B se ztratí z
  notifikace

  B zůstane v dirty_idx s flagem false, takže consumer ho vyzvedne až po dalším
  publish() — o 16 ms později. Renderer mezitím drží starý stav o tick déle →
  stateDelta vyjde 33 ms místo 16 → viditelné škubnutí. Okno je úzké, ale
  nenulové. Čistý fix je flag zahodit a „je tu novinka" zakódovat jako bit přímo
  do dirty_idx, aby výměna byla jedna atomická operace.

  Catch-up krok zahodí stav i razítko

  Když dispatchAction udělá v jedné iteraci dva kroky, oba zavolají scenePublish
  — druhý ten první přepíše (což je v pořádku, triple buffer drží jen
  poslední). Ale obě sinceStart() volání se liší o mikrosekundy, takže razítko
  odpovídá jednomu kroku, zatímco fyzika popojela o 2Δ. Renderer pak spadne do
  fallbacku stateDelta <= 0.0001 (Renderer.h:211) a alfa je mimo. Monotónní
  simulační hodiny řeší i tohle.

  Render smyčka není nijak brzděná

  V runLoop je timerSyncV.waitUntilNextStep() zakomentovaný (Renderer.h:241),
  takže render vlákno běží naplno a přes tick() + prázdný dispatchAction jen
  protáčí akumulátor. Logic vlákno přitom v waitUntilNextStep spinuje s 
  _mm_pause() až 2 ms každých 16,6 ms (TimeManager.h:249-255) na
  THREAD_PRIORITY_HIGHEST. Na stroji s málo jádry si ta dvě vlákna lezou do zelí
  — a jitter scheduleru jde, viz bod 1, rovnou do interpolace.

  timer.tickAndFlush() na Renderer.h:196 je mimochodem skoro no-op: renderer z
  toho timeru používá jen sinceStart() a getFixedDelta(), a sinceStart() si dělá
  vlastní QPC nezávisle na akumulátoru. Reálně to jen udržuje FPS historii.

  Dvě drobnosti, které se taky projeví

  Animace jede na indexu snímku. CreateGridWave(60, 0.2f, frameIndex, 0.05f)
  předává frameIndex do parametru float time — proto ten warning C4244 na
  Renderer.h:128 v buildu. Rychlost vlny tedy závisí na FPS; jakmile FPS kolísá,
  vlna kolísá s ním.

  Garbage v prvním publikovaném stavu. std::array<T,3> buffers; se

     Odstranění jitteru z interpolace (worktree remove_jitter)

     Context

     Logic vlákno běží na pevných 60 Hz a publikuje LogicState render vláknu
     přes triple buffer. Každý stav nese razítko tickInfo.lastLogicTick a
     renderer podle něj interpoluje mezi předchozím a aktuálním stavem.

     Problém: TimeManager::dispatchAction (dvouargumentová varianta,
     TimeManager.h:123-129) předává jako razítko sinceStart() — tedy syrový QPC
     v okamžiku, kdy se callback náhodou spustil. Razítka jsou proto k·Δ + 
     jitter_k. Renderer pak počítá stateDelta = timeCurr - timePrev a t = 
     (visualTime - timePrev) / stateDelta (Renderer.h:206-220), takže jitter
     scheduleru jde přímo do interpolačního koeficientu. Alfa neroste lineárně s
     reálným časem → mikro-stutter i při vysokém FPS. Diagnostický výpis
     "Zaskub: t = ..." na Renderer.h:222 je symptom právě tohohle.

     Druhý projev: když dispatchAction udělá v jedné iteraci dva kroky
     (catch-up), sinceStart() se volá uvnitř smyčky a oba kroky dostanou skoro
     identické razítko. stateDelta spadne k nule, renderer sáhne po fallbacku
     stateDelta <= 0.0001 (Renderer.h:211), zatímco fyzika popojela o 2Δ → alfa
     je mimo.

     Cílem je razítko, které je rovnoměrné (bez jitteru) a zároveň zůstane ve
     stejných hodinách, jaké používá renderer.

     Rozsah dopadu (ověřeno)

     Dvouargumentový dispatchAction má v celém živém kódu jediného konzumenta:
     App.h:523. Jednoargumentová varianta (Renderer.h:245, Renderer.h:233,
     App.h:529) se nemění. getInterpolationFactor() a getAccumulator() nikdo
     nevolá.

     Klíčová myšlenka

     Nepoužít čítač m_simTime += Δ. Ten by se od nástěnných hodin trvale rozešel
     při každém zásahu clampu v tick() (TimeManager.h:98, limit = 0.25 s), a
     protože renderer počítá visualTime z vlastního sinceStart(), po jednom
     velkém záseku by se interpolace rozbila natrvalo.

     Místo toho odvodit ideální čas z akumulátoru. m_prev je vždy okamžik, ke
     kterému akumulátor odpovídá (aktualizují se spolu jak v tick(), tak v
     updateAccumulatorOnly()). Platí invariant A = accumulated - (k-1)·Δ při
     nástěnném čase W = m_prev. Po odečtení A' = A - Δ dostaneme:

     W - A' = W₀ + k·Δ + (zahozený čas)

     Tedy razítka přesně Δ od sebe, ukotvená v nástěnných hodinách, a každý krok
     v catch-upu dostane své vlastní správné razítko. Clamp způsobí jen
     jednorázový posun offsetu (což je správně — ten čas jsme zahodili záměrně),
     ne trvalý rozjezd.

     Protože obě TimeManager instance vznikají s useGlobalStart=true (App.h:492,
     Renderer.h:168), sdílejí m_absoluteStart a razítka jsou napříč vlákny
     porovnatelná. Tenhle epoch se nesmí rozbít, jinak Renderer.h:217-220 tiše
     spočítá věrohodné, ale špatné t.

     Změny

     1. GLibpp/src/Utility/TimeManager.h — jádro fixu

     Přidat privátní helper vedle updateAccumulatorOnly():

     // Nastenny cas (sekundy od spolecneho epochu), kteremu odpovida soucasny 
     m_accumulator.
     // m_prev a m_accumulator se vzdy aktualizuji spolecne (tick(), 
     updateAccumulatorOnly()).
     double accumulatorTimeInSeconds() const {
         return double(m_prev.QuadPart - m_absoluteStart.QuadPart) /
     double(m_freq.QuadPart);
     }

     Přepsat dvouargumentový dispatchAction:

     void dispatchAction(const std::function<void(double, double)>& 
     stepCallback)
     {
         if (m_accumulator < m_fixedDelta) return;

         // Nesmi to byt cerstve sinceStart() - to by do razitka vratilo mezeru
         // mezi tick() a timhle volanim. Akumulator se vztahuje k m_prev.
         const double wallAtAccumulator = accumulatorTimeInSeconds();

         while (m_accumulator >= m_fixedDelta) {
             m_accumulator -= m_fixedDelta;
             // wallAtAccumulator - m_accumulator == W0 + k*fixedDelta
             // = idealni cas KONCE tohoto kroku, rovnomerne, bez jitteru.
             stepCallback(m_fixedDelta, wallAtAccumulator - m_accumulator);
         }
     }

     Dvě věci, které to řeší mimochodem:
     - Odečtení Δ před callbackem znamená, že razítko je čas konce kroku.
     Publikovaný stav je stav po updateLogic(), takže tím zaniká i současný
     posun o jedno Δ (dnes se razítko bere před krokem, ale stav se publikuje po
     něm).
     - sinceStart() se přestane volat na každý krok (jedno QPC navíc na krok).

     tickAndFlush(), jednoargumentový dispatchAction ani tick() se nemění.

     2. GLibpp/src/Engine/Renderer/Renderer.h — konzument

     - Lag musí být přesně 1.0Δ (Renderer.h:217). V commitu je * 1.5, změnit na
     * 1.0. Není to kosmetika: se správnými razítky platí timeCurr = now - r,
     kde r ∈ [0,Δ), takže t = r/Δ ∈ [0,1). Při *1.5 vyjde t = r/Δ - 0.5 ∈ [-0.5,
     0.5) → půlku času záporné → clampne se na 0 → stutter.
     - Vrátit clamp na [0,1] a výpis rozšířit na obě strany, aby byl skutečným
     detektorem:
     if (t > 1.0 || t < 0.0) std::cout << "Zaskub: t = " << t << std::endl;
     double tClamped = std::clamp(t, 0.0, 1.0);
     - Clamp 1.1 v pracovní kopii je náplast, kterou tenhle fix nahrazuje.
     - Frame limiter nechat vypnutý (dle rozhodnutí): zakomentovat
     timerSyncV.waitUntilNextStep() na Renderer.h:241. Commit má limiter zapnutý
     na 35 Hz, což by strop render FPS pod logiku.

     3. GLibpp/src/App/Scene.h — inicializace razítka

     LogicTickInfo::lastLogicTick nemá inicializátor a std::array<T,3> buffers v
     triple bufferu se default-inicializuje, takže publish() na App.h:510 (před
     prvním tickem) pošle neinicializovaný double. Projeví se to jako glitch na
     prvních snímcích.

     struct LogicTickInfo {
         double lastLogicTick = 0.0;
     };

     4. GLibpp/src/App/App.h — beze změny

     Lambda na App.h:523 už parametr lastLogicTick přijímá a předává do
     scenePublish. Sémantika se mění pod ní, kód ne.

● Hotovo, zacommitováno na větvi worktree-remove_jitter.

  Co fix dělá

  Razítko logického kroku se místo sinceStart() odvozuje z akumulátoru: S = 
  wallAt(m_prev) - m_accumulator. Protože tick() i updateAccumulatorOnly()
  posouvají m_prev i akumulátor o stejné dt, S zůstává vůči jitteru scheduleru
  imunní a každý krok ho posune přesně o Δ — přitom zůstává ukotvené na
  nástěnných hodinách, takže se po záseku nerozejde s rendererem.

  Odečet akumulátoru před callbackem je nosný: díky němu razítko patří konci
  kroku, tedy přesně tomu stavu, který se publikuje.

  Naměřená čísla
